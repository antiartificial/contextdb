# v0.59.0 Release Recap

v0.59.0 is a non-breaking durability release that adds dry-run-first KV hot-key refresh repair.

## What Changed

| Area | New in v0.59.0 | Why it matters |
|:-----|:---------------|:---------------|
| KV refresh plan | `contextdb repair kv-cache --key KEY --value VALUE --report` | Lets operators review exactly which hot keys would be refreshed |
| Reviewed execution | `--execute` | Writes only after an explicit operator opt-in |
| Value sources | `--value` and `--value-file PATH` | Supports literal values and reviewed cache payload files |
| Safety controls | `--overwrite`, `--ttl` | Existing keys are skipped by default; TTL is explicit when needed |
| Feature metadata | `kv-cache-refresh-repair` | Lets live deployments advertise reviewed KV cache refresh support |

## Compatibility

This release only adds an opt-in local repair command. Existing doctor checks, KV reads, cache contents, REST APIs, storage formats, and SDK contracts are unchanged unless an operator runs `--execute`.

## Operator Flow

```bash
CONTEXTDB_DATA_DIR=/var/lib/contextdb \
  contextdb repair kv-cache --key context:my-app:active --value-file ./active-context.json --report

CONTEXTDB_DATA_DIR=/var/lib/contextdb \
  contextdb repair kv-cache --key context:my-app:active --value-file ./active-context.json --execute --report
```

Pair this with `contextdb doctor --kv-key KEY`: doctor identifies missing hot keys, then `repair kv-cache` plans and executes reviewed refreshes.
